# REVISION – Reflexión final del proyecto

## Conflictos y cómo los resolvimos
Durante el desarrollo del proyecto surgieron varios conflictos de Git, especialmente en la Sesión 3 cuando ambos trabajamos en ramas separadas y modificamos archivos relacionados.  
Los conflictos se resolvieron utilizando:

- `git pull` para traer cambios remotos antes de continuar.
- `git rebase` para reordenar commits y limpiar el historial.
- Resolución manual de conflictos en archivos Markdown.
- Revisión cruzada mediante Pull Requests.

El uso de ramas claras y commits descriptivos facilitó la resolución.

## Comandos Git más utilizados
Los comandos más usados durante el proyecto fueron:

- `git clone`  
- `git checkout -b`  
- `git add`  
- `git commit`  
- `git push`  
- `git pull`  
- `git rebase`  
- `git merge`  
- `git log`  
- `git status`  

---

## Qué haríamos diferente en un próximo proyecto
- Crear ramas más pequeñas y específicas para evitar conflictos grandes.  
- Hacer commits más frecuentes y atómicos.  
- Revisar los PRs antes de acumular demasiados cambios.  
- Definir desde el principio una estructura de carpetas más estricta.  
- Documentar los cambios de alcance en cuanto ocurren.

---

## Funcionamiento del intercambio de roles
El intercambio de roles funcionó correctamente y permitió:

- Comprender mejor el trabajo del otro miembro.  
- Detectar inconsistencias en la documentación.  
- Mejorar la calidad final al revisar desde otra perspectiva.  
- Repartir la carga de trabajo de forma equilibrada.

El cambio de rol también reforzó el uso de Git, ya que ambos tuvimos que trabajar en partes del proyecto que inicialmente no habíamos creado.

---

## Conclusión
El proyecto se completó con éxito, cumpliendo todas las sesiones, integrando el cambio de alcance y generando una documentación coherente, estructurada y funcional.  
El repositorio está listo para el release v1.0.
